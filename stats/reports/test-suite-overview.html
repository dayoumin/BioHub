테스트 개념 정리

Vitest

이 프로젝트의 기본 테스트 러너입니다. 빠르게 실행되는 단위 테스트, 훅 테스트, 컴포넌트 테스트, 회귀 테스트 대부분이 여기에 들어갑니다. 보통 pnpm test가 이 범위를 뜻합니다.

Playwright

실제 브라우저를 띄워서 사용자의 클릭 흐름을 확인하는 E2E 도구입니다. Vitest보다 느리지만, “페이지가 진짜로 움직이는지”를 보는 데 필요합니다.

단위 테스트

함수 하나, 작은 컴포넌트 하나가 의도대로 동작하는지 확인합니다. 실패 원인을 좁게 잡을 수 있어서 가장 자주 실행하는 테스트입니다.

회귀 테스트

예전에 고친 버그가 다시 나오지 않게 잠그는 테스트입니다. 파일명에 bugfix, regression, scenario가 많이 붙습니다.

통합 테스트

업로드, method 선택, variable mapping, executor 같은 여러 층을 같이 묶어 봅니다. 단위 테스트보다 느리지만 실제 사용자 흐름에 더 가깝습니다.

계약 테스트

모듈 간 약속이 깨지지 않는지 검증합니다. 예를 들어 method id, selector mapping, schema, variable requirements 같은 “인터페이스”를 고정하는 역할입니다.

Golden Value 테스트

계산 결과를 Python/R 기준값과 비교하는 테스트입니다. 통계 앱에서는 UI 테스트보다 더 중요할 때가 많습니다. “보기에 맞는지”가 아니라 “숫자가 맞는지”를 확인합니다.

Validation Runner

단일 테스트 파일이 아니라, 별도 스크립트로 여러 메서드를 기준값과 대조하는 검증 계층입니다. 일반 Vitest와 달리 정확도 중심의 검증 축으로 이해하면 됩니다.

Coverage

코드가 얼마나 테스트를 통해 실행됐는지를 보는 지표입니다. 품질을 보조하는 숫자일 뿐, coverage가 높다고 정확도까지 보장되는 것은 아닙니다.

Skipped 테스트

일시적으로 건너뛴 테스트입니다. 전체 실행에서 숫자는 보이지만 실제 검증은 하지 않습니다. 누적되면 품질 부채가 되므로 주기적으로 정리해야 합니다.

적절성 체크

적절핵심 리스크에 테스트가 몰려 있음 `lib(87)`, `components(56)`, `services(29)`에 가장 많이 분포합니다. 이 프로젝트에서 실제 장애가 자주 나는 계산 로직, selector 매핑, 실행기 연결, 복잡한 UI 흐름에 테스트가 집중된 구조입니다.
적절테스트 층이 분리돼 있음 Vitest 단위/회귀 테스트, Playwright E2E, golden value, validation runner가 따로 있어 “빠른 피드백”과 “정확도 검증”이 구분되어 있습니다.
정리 필요디렉터리 의미가 섞여 있음 `bugfix`, `simulation`, `session-c`, `graph-studio` 같은 폴더는 역사적 맥락은 남기지만, 신규 기여자가 “어디에 테스트를 추가해야 하는지” 판단하기 어렵게 만듭니다.
정리 필요실행 정책 문서화가 부족함 어떤 테스트를 “매일”, “PR 전”, “야간 검증”으로 돌려야 하는지 코드만 보고 알기 어렵습니다. 운영 기준을 별도로 정리해 두는 편이 좋습니다.

추천 운영 원칙

  • 새 버그를 고칠 때는 일반 테스트보다 먼저 회귀 테스트 1개를 추가하고 수정에 들어갑니다.
  • `contracts`, `registry`, `validation`, `golden-values`는 “정확도 방어선”이므로 CI에서 빠지면 안 됩니다.
  • `bugfix`, `simulation` 폴더는 당장 없애기보다, 새 테스트부터는 기능 소유 폴더로 옮겨 적재하는 편이 안전합니다.
  • 매일 전체 스위트를 전부 돌리기보다, 로컬은 타깃 실행, PR/merge는 전체 실행으로 역할을 나누는 편이 효율적입니다.
한 줄 판단
지금 테스트 수가 “과하게 많은 것”이라기보다, 복잡한 통계 앱에 필요한 검증 축이 많아서 생긴 결과에 가깝습니다. 다만 폴더 네이밍과 실행 정책은 더 명확하게 다듬을 여지가 있습니다.
