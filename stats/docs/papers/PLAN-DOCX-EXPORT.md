# Phase 4: DOCX 내보내기 설계

**일자**: 2026-04-02
**상태**: 승인 대기
**선행**: Phase 3 (Plate 리치 텍스트 에디터) 완료

---

## 1. 목표

DocumentBlueprint를 학술 논문 형식의 DOCX 파일로 내보내기.
DocumentExportBar의 비활성 DOCX 버튼을 활성화.

## 2. 설계 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| 저널 스타일 | 기본 1종 (일반 학술) | 저널별 미세 조정은 Word에서 직접. 구조화된 DOCX가 깨끗하게 나오는 것이 핵심 |
| 콘텐츠 소스 | `content` (마크다운) + export 직전 flush | DocumentEditor가 500ms 디바운스로 serialize. export 전에 강제 flush 해야 최신 반영 |
| Figure 처리 | 캡션 플레이스홀더만 | 이미지 삽입은 Phase 6b (오프스크린 렌더링) 에서 |
| 테이블 빌더 | 독립 구현 | 기존 `docx-export.ts`는 보고서 스타일 (파란 헤더, 음영). 논문은 3-line table |
| 파일명 | `${doc.title}.docx` | 기존 HTML 내보내기 패턴과 일관 (`${doc.title}.html`) |
| 날짜 | 본문: `updatedAt`, 푸터: export 시각 | 본문=마지막 수정일, 푸터=출력 시점 |
| 빈 섹션 판정 | `content \|\| tables?.length \|\| figures?.length` | `documentToHtml`의 `hasContent` 로직과 동일 |
| DOCX/HTML 순회 공유 | `hasVisibleContent()` 헬퍼 1개만 | 출력 타입이 달라 전체 visitor 패턴은 과잉. 4번째 포맷 시 재검토 |

## 3. 스타일 상수

```
폰트: Times New Roman
  - 본문: 12pt (24 half-pt)
  - 제목 (H1): 14pt (28 half-pt), bold, 중앙정렬
  - 섹션 헤딩 (H2): 12pt, bold
  - 서브헤딩 (H3): 12pt, bold
줄간격: 2.0 (480 twips, 더블스페이스)
마진: 1인치 (1440 twips) 전방향
표: 3-line table (상단선 + 헤더 구분선 + 하단선, 세로선 없음)
캡션: Table/Figure — 이탤릭
페이지 번호: 하단 중앙
```

## 4. DOCX 문서 구조

```
[제목] — Heading 1, 중앙정렬
[저자] — 중앙정렬, 쉼표 구분
[날짜] — updatedAt (중앙정렬, 이탤릭, 회색)

[각 섹션 반복] (hasVisibleContent 통과한 섹션만)
  섹션 제목 — Heading 2
  본문 — 마크다운 줄 단위 파싱 → Paragraph + TextRun
  표 — section.tables[] → 3-line table
  그림 — section.figures[] → "Figure N: 캡션" 이탤릭 플레이스홀더

[푸터] — "Generated by BioHub" + export 시각, 페이지 번호
```

## 5. 마크다운 파싱 범위

HTML 내보내기(`documentToHtml`)와 동일 수준:

| 마크다운 | DOCX 변환 |
|---------|----------|
| `## H2` | HeadingLevel.HEADING_2 |
| `### H3` | HeadingLevel.HEADING_3 (또는 bold paragraph) |
| `**bold**` | TextRun `{ bold: true }` |
| `*italic*` | TextRun `{ italics: true }` |
| `\n\n` (빈 줄) | 단락 구분 |

리스트, 인용, 인라인 코드, 중첩 마크(`***bold italic***`)는 Phase 4 범위 밖 (후속 개선).
중첩 마크 입력 시 외측 마크만 적용 (데이터 손실 없음, 스타일 일부 누락).

## 6. Export 직전 serialize flush

**문제**: `DocumentEditor`는 `plateValue`를 즉시 저장하지만 `content`(마크다운) 직렬화는 500ms 디바운스. 사용자가 입력 직후 내보내기를 누르면 최신 본문이 DOCX에 빠질 수 있음.

**해결**:
1. `DocumentEditor`에서 `flushSerialize()` 함수를 `DocumentExportBar`에 `onBeforeExport` prop으로 전달
2. DOCX/HTML/마크다운 모든 내보내기 핸들러에서 `await onBeforeExport()` → flush 후 최신 `doc` state로 export
3. `flushSerialize`는 이미 구현됨 (`DocumentEditor.tsx:121`) — 섹션 전환 시 호출하는 것과 동일

**변경 범위**:
- `DocumentExportBar`: `onBeforeExport?: () => void` prop 추가
- `DocumentEditor`: `flushSerialize`를 `DocumentExportBar`에 전달
- HTML/마크다운 내보내기도 동일하게 flush 호출 (일관성)

## 7. 파일 변경

### 새 파일 (1개)

**`stats/lib/services/export/document-docx-export.ts`**

```typescript
// 순수 빌더 (테스트 가능)
export async function buildDocxDocument(doc: DocumentBlueprint): Promise<Document>

// 사이드이펙트 래퍼 (다운로드 실행)
export async function documentToDocx(doc: DocumentBlueprint): Promise<void>

// 공유 헬퍼
export function hasVisibleContent(section: DocumentSection): boolean
```

- `docx` 패키지 dynamic import (기존 패턴)
- `downloadBlob()` 재활용 (`export-data-builder.ts`)
- 스타일 상수는 파일 상단에 정의 (별도 파일 불필요 — 프리셋 1종)
- 내부 함수:
  - `buildAcademicTable(table: DocumentTable, docx)` — 3-line table
  - `parseMarkdownLine(line: string, docx)` — 단일 줄 → Paragraph
  - `parseInlineMarks(text: string, docx)` — bold/italic 인라인 파싱 → TextRun[]

## 10. 재활용 vs 독립 구현

| 항목 | 소스 | 방식 |
|------|------|------|
| `downloadBlob()` | `export-data-builder.ts` | import 재활용 |
| `docx` dynamic import | `docx-export.ts` 패턴 | 패턴만 참조, 독립 작성 |
| 테이블 빌더 | 신규 | 학술 3-line table (기존과 스타일 다름) |
| 마크다운 파싱 | `documentToHtml` 참조 | 동일 수준 독립 구현 |
| 색상/스타일 상수 | 신규 | 학술 논문용 (기존 보고서용과 다름) |
| `hasVisibleContent` | 신규 export | DocumentExportBar에서도 import (빈 문서 체크) |
