# 코드 접근 관리 & 누출 방지 전략

> 외주/팀 확장 시 코드 접근 부여와 보안 관리 가이드

---

## 1. GitHub 권한 관리

### 권한 레벨

| 레벨 | 권한 | 대상 |
|------|------|------|
| **Read** | 코드 열람만 | 리뷰어, 참고용 |
| **Triage** | 이슈 관리 | PM, QA |
| **Write** | 푸시 가능 | 개발자 |
| **Maintain** | 설정 일부 변경 | 팀 리드 |
| **Admin** | 전체 | CTO, 핵심 인원 |

### 구조적 분리

**멀티레포 (대기업 표준)**
- 프론트엔드 / 백엔드 / 인프라를 별도 레포로 분리
- 외주에게 해당 레포만 접근 부여
- 예: `biohub-frontend`, `biohub-api`, `biohub-infra`

**모노레포 + CODEOWNERS**
- 디렉토리별 PR 리뷰 승인 강제 가능
- 단, 읽기는 전체 가능 → 진정한 코드 격리는 안 됨

```
# CODEOWNERS 예시
/infra/          @core-team
/.github/        @core-team
/src/api/        @backend-team
/src/components/ @frontend-team
```

### 외주/계약 개발자에게 흔히 쓰는 방법

- **별도 레포 생성** — 필요한 부분만 추출한 레포를 만들어 제공 (포크 아님 — 포크는 원본 연결이 남음)
- **GitHub Teams** — 팀별로 레포 접근 범위 설정
- **Fine-grained PAT** — 특정 레포만 접근 가능한 토큰 발급 (2023년 도입, classic PAT보다 세밀한 권한 제어)
- **GitHub Enterprise + Repository Rules** — 브랜치 보호, 시크릿 접근 제한

### 규모별 관행

| 규모 | 전략 |
|------|------|
| 스타트업 (5명 이하) | 전체 접근 + 브랜치 보호만 |
| 중소 (10~50명) | 멀티레포 + 팀별 접근 |
| 대기업 | GitHub Enterprise + SSO + 감사 로그 |
| 외주 포함 | 별도 레포 or Organization 내 제한된 Team |

## 2. 코드 누출 방지

### 법적 보호 (가장 기본이자 핵심)

| 계약서 | 역할 |
|--------|------|
| **NDA (비밀유지계약)** | 코드 유출 시 법적 책임 |
| **업무위탁계약서** | 산출물 저작권 귀속 명시 |
| **경업금지조항** | 유사 서비스 개발 제한 (기간 한정) |

> 한국 실무 표준: **NDA + 업무위탁계약** — 외주든 내부 직원이든 입사/계약 시 서명

### 기술적 통제

**접근 감사 (Audit)**
- GitHub Enterprise: 누가 언제 어떤 파일을 clone/download했는지 로그
- 무료 플랜에서도 `git log`, PR 히스토리로 활동 추적 가능

**시크릿 분리 (필수)**
- `.env` 파일은 절대 레포에 포함하지 않음
- GitHub Environments로 시크릿을 환경별 분리
- 외주 개발자는 `staging` 환경만 접근, `production`은 내부만

**접근 범위 최소화**
- 필요한 브랜치만 접근 (branch protection rules)
- Deploy key는 환경별로 분리
- 계약 종료 시 즉시 접근 권한 회수

**코드 자체의 방어**
- 핵심 비즈니스 로직을 API 서버 뒤에 숨기기 (프론트엔드만 외주)
- 민감한 알고리즘은 별도 private 패키지로 분리

### 현실적 인식

- **코드 유출은 100% 막을 수 없다** — 대기업도 마찬가지
- 코드 자체보다 **시크릿(API 키, DB 토큰)과 배포 권한**을 제한하는 것이 핵심
- 내부 개발자에게도 전체 코드 접근을 주는 것이 일반적
- 경쟁력은 코드가 아니라 **실행력, 데이터, 사용자, 도메인 전문성**

## 3. BioHub 적용 계획

### 단기 (외주 1~2명)

- [ ] GitHub Organization 생성 (무료)
- [ ] Team 만들어서 레포 `Write` 권한 부여
- [ ] 브랜치 보호 규칙 설정 (main 직접 푸시 금지, PR 필수)
- [ ] `.env` 관련 시크릿은 별도 전달 (staging 전용)
- [ ] CODEOWNERS 파일로 핵심 경로 PR 리뷰 강제
- [ ] **NDA + 업무위탁계약 체결**

### 중기 (팀 확장 시)

- [ ] 프론트엔드 / 통계엔진(Pyodide workers) / 인프라를 분리 레포로 전환
- [ ] API 계약(타입 정의)만 공유 패키지로 관리
- [ ] GitHub Enterprise 또는 SSO 도입 검토
