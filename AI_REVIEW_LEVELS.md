# AI 리뷰 요청 템플릿

AI 리뷰를 받을 때마다 요구 범위가 들쭉날쭉하면 “어디까지 검토했는지” 정리하기 어렵습니다. 아래 템플릿은 리뷰 깊이를 네 가지 단계로 나눠 미리 약속해 두는 용도입니다. 새 작업을 검토할 때는 **레벨 + 대상(파일/기능) + 테스트 로그**만 전달하면 일관된 품질을 유지할 수 있습니다.

---

## 1. 리뷰 레벨 정의

| 레벨 | 초점 | 포함 항목 | 제외 항목 |
| --- | --- | --- | --- |
| **Level 0 – 테스트 리포트** | 실행 결과 확인 | `tsc`, 핵심 `npm test`, lint/FMT 로그 확인, 실패 원인만 요약 | 코드 구조/스타일 리뷰, UI 피드백 |
| **Level 1 – 부분 코드 리뷰** | 특정 파일/함수 정적 분석 | 지정 파일의 타입·런타임 이슈, 명백한 사이드이펙트, 빠진 예외 처리 | 다른 파일, UI/UX, 접근성 |
| **Level 2 – 기능 흐름 리뷰** | 기능 단위 점검 | 신규 흐름 전체, 폴백/에러 처리, 회귀 위험, 관련 테스트 커버리지 | UI 미세 스타일, 번역, 문서 |
| **Level 3 – 풀 리뷰** | 배포 직전 전체 점검 | Level 2 범위 + UI/접근성, 카피/번역, 문서, 추적 로그, 성능 이슈 | - |

> ⚖️ 기본값은 Level 1입니다. 별도 언급이 없으면 지정된 파일만 정적 리뷰로 간주합니다.

---

## 2. 요청 템플릿

```text
Subject: [리뷰 Level X] <기능 또는 파일 이름>

1. 작업 요약
   - <주요 변경 사항 2~3줄>

2. 확인 범위
   - Level X
   - 대상: <파일/폴더 명시>

3. 테스트 로그
   - `npx tsc --noEmit` ✅
   - `npm test <suite>` ✅/❌ (실패 시 로그 발췌)
```

필요하면 “이전 리뷰에서 통과한 파일은 제외”, “UI 경고 메시지 문구만 검토”처럼 추가 조건을 *확인 범위* 항목에 적어 주세요.

---

## 3. AI 응답 형식 (권장)

1. **Findings** – 심각도 순으로 bullet (파일:라인)
2. **Tests** – 전달받은 로그 재확인 + 추가로 돌린 항목
3. **Next Steps** – 수정 제안 또는 남은 TODO

Level 0에서는 Findings 대신 “테스트 결과”만 요약하고, Level 3에서는 접근성/UX 항목을 따로 묶어도 좋습니다.

---

## 4. 예시

```
[리뷰 Level 2] PurposeInputStep 하이브리드 흐름
- Level 2 / 대상: components/smart-flow/steps/PurposeInputStep.tsx
- 테스트: npx tsc --noEmit ✅, npm test purpose-input-step ❌ (act 경고)
```

이렇게 요약한 뒤 AI에게 “위 템플릿 기준으로 리뷰해줘”라고만 요청하면, 매번 범위를 길게 설명하지 않아도 됩니다. 필요 시 이 문서를 업데이트하며 팀 표준을 확장하세요.
