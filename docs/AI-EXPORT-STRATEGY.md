# AI Export 전략: 연구 패키지 체계화

**Last updated**: 2026-04-04

**정기 점검**: 분기별 (다음: 2026-07) — 경쟁 환경, AI 모델 동향, 포지셔닝 유효성 확인

## 1. 핵심 방향

BioHub은 논문을 직접 쓰는 도구가 아니다.
**AI 모델(Gemini, ChatGPT, Claude)에 넘길 "연구 패키지"를 체계화하는 플랫폼**이다.

---

## 2. 3단계 논문 완성 모델

```
┌─────────────────────────────────────────────────────────┐
│  1차 BioHub (Gemini 2.5 Flash Lite)                     │
│  ─────────────────────────────────────                   │
│  • 검증된 라이브러리로 통계/그래프/계산 실행              │
│  • 결과 체계화: 파일, 그림, 표, 목차 정리                │
│  • AI-ready 패키지 export (Markdown/JSON)                │
│                                                         │
│  내부 AI 역할: 분석 추천, 결과 해석 보조, 차트 patch     │
│  딥리서치/논문 생성 안 함 → 비용 효율 극대화             │
├─────────────────────────────────────────────────────────┤
│  2차 사용자의 SOTA AI (Claude/GPT/Gemini Pro)            │
│  ─────────────────────────────────────                   │
│  • BioHub 패키지를 받아서 논문 초고 생성                 │
│  • 딥리서치, 문맥 이해, 학술 문체 적용                   │
│  • 사용자가 이미 구독 중인 모델 활용                     │
├─────────────────────────────────────────────────────────┤
│  3차 사람 (Word/HWP)                                     │
│  ─────────────────────────────────────                   │
│  • 학회 템플릿(HWPX/DOCX/LaTeX)에 맞춰 최종 편집        │
│  • 한국 학회 → HWP/HWPX, 국제 저널 → DOCX/LaTeX         │
│  • 제출                                                 │
└─────────────────────────────────────────────────────────┘
```

### 왜 BioHub이 직접 논문을 쓰지 않는가

- **비용**: Flash Lite로 체계화만 하면 API 비용 최소. 딥리서치급 작업은 사용자의 SOTA 모델에 위임
- **품질**: 논문 생성은 최고 성능 모델이 해야 품질이 나옴. BioHub이 중급 모델로 하면 오히려 가치 하락
- **유연성**: 사용자가 선호하는 AI를 자유롭게 선택 (Claude, GPT, Gemini 등)

### 왜 AI가 직접 통계를 대체할 수 없는가

SOTA AI가 코드를 짜도 **1%의 오류가 학술에서는 치명적**:
- `ttest_ind` vs `ttest_rel` 혼동 → p값 오류
- 자유도 계산 실수 → reviewer가 잡아냄
- 그래프 축 스케일 오류 → 결론 왜곡

연구자가 매번 AI 코드를 검증하는 비용 > BioHub의 검증된 라이브러리를 쓰는 비용.
**BioHub = hallucination 없는 분석 레이어**.

## 4. 보완 필요 영역 (우선순위순)

### 4-1. AI Export 기능 (1순위 — 핵심 차별화)

프로젝트의 모든 결과를 **하나의 구조화된 패키지로 내보내기**. 상세 설계: [PLAN-PAPER-PACKAGE-ASSEMBLY.md](PLAN-PAPER-PACKAGE-ASSEMBLY.md)

핵심 설계 결정 (검토 반영):
- **하이브리드 포맷**: 지시사항/맥락 = Markdown, 분석 결과 = JSON (숫자 정밀도 보존)
- **방어적 프롬프트**: "수치 변경 금지", "날조 금지" 등 anti-hallucination 규칙 내장
- **문헌 요약 필수**: 요약 없이는 AI가 서론을 hallucination으로 채움
- **그래프 = 텍스트 패턴 요약 기본**: base64 제거, 이미지는 선택적 별도 첨부
- **검증 체크리스트 자동 생성**: 핵심 수치 대조표를 패키지 하단에 포함
- **5단계 UX**: 연구 개요 → 결과 배치+태깅 → 문헌 → 저널+맥락 → Export
- **MVP에서도 end-to-end 가능**: Citation Store 없이 BibTeX/수동 입력으로 작동

**이것이 없으면 BioHub은 "통계 도구 모음"에 불과하고, 이것이 있으면 "AI 시대 연구 허브"가 된다.**

### 4-2. 결과 ↔ 논문 섹션 매핑 (2순위)

- 각 분석 결과에 논문 섹션 태그 부여: `Table 2`, `Figure 3`, `Results 3.1`
- 논문 구조(Introduction → Methods → Results → Discussion)에 자동 배치
- AI에게 넘길 때 "이 데이터는 Results 섹션용"이라는 맥락 자동 포함

### 4-3. 프롬프트 템플릿 (3순위)

- 분야별 논문 작성 프롬프트 생성기
  - "이 통계 결과로 Results 섹션을 APA 스타일로 작성해줘"
  - "이 문헌 목록을 기반으로 Introduction의 선행연구 검토를 작성해줘"
- 생태학, 유전학, 수산학 등 분야별 템플릿
- 저널별 스타일 가이드 반영

### 4-4. Publication-ready 그래프 Export (4순위)

- 저널 투고용 포맷: TIFF 300dpi, EPS, SVG
- APA/학술 스타일 자동 적용 (축 라벨, 폰트 크기, 범례 위치)
- Figure 캡션 자동 생성 (분석 메타데이터 기반)

### 4-5. 버전/히스토리 비교 (5순위)

- 분석 조건 변경 시 결과 차이 추적
- Reviewer 코멘트 대응: "분석 조건을 X로 바꾸면 결과가 어떻게 바뀌는지" 즉시 비교
- 분석 재현성 보장

## 5. 시장성 평가

### 긍정 요인

- **타이밍**: 연구자의 AI 활용 폭발적 증가 (2024~2026)
- **공백**: 기존 통계 SW(SPSS, R, JASP)는 AI 연동 없음
- **블루오션**: "AI-ready research workflow" 카테고리 자체가 신규
- **비용 우위**: Cloudflare 기반 → 낮은 운영비, 무료/저가 진입 가능
- **프라이버시**: 브라우저 내 계산 → 민감한 연구 데이터 외부 전송 불필요

### 위험 요인

- **AI 직접 통계**: ChatGPT/Claude가 통계+그래프 기능을 강화하면 중간 레이어 가치 감소
  - 단, **1% 오류도 허용 안 되는 학술 영역**에서 검증된 라이브러리 기반 BioHub의 신뢰성이 해자
  - 도메인 특화 워크플로우 + 히스토리 + 체계화 + 재현성은 범용 AI가 대체 어려움
- **타겟 집중 필요**: "모든 연구자" → **"생물학/생태학/수산학 연구자"**로 좁혀야 함

### 왜 BioHub의 포지션이 비어 있는가

기존 도구들은 **한 가지만 잘한다**:

```
통계 도구 (JASP, SPSS)     → 분석은 하지만 AI 연동 없고 export는 복붙
AI 작성 도구 (Paperpal 등) → 글은 쓰지만 통계 엔진 없음
코드 기반 (Quarto)         → 둘 다 하지만 코딩 필수 → 비전공자 진입장벽
문헌 관리 (Zotero 등)      → 레퍼런스만, 분석/작성 없음
```

**"GUI로 통계 → 결과 체계화 → AI가 소비 가능한 패키지 export"를 하는 도구는 현재 없다.**

이 공백이 존재하는 이유:
1. 통계 도구 회사(IBM, GraphPad)는 AI 연동에 관심 없음 (레거시 수익 모델)
2. AI 작성 도구(Paperpal 등)는 통계 엔진을 만들 역량/의지 없음
3. Quarto/R Markdown은 코딩 전제라 GUI 시장과 다른 세그먼트
4. 한국 시장은 HWP 중심이라 글로벌 도구가 진입 안 함

**BioHub의 "검증된 분석 + 체계화 + AI Export" 조합은 현재 직접 경쟁자 없음.**

### 끼어들 수 있는 경쟁 지점

| 구간 | 위협 후보 | 위협도 | 이유 |
|------|----------|--------|------|
| 1차 (분석) | AI Code Interpreter | 중 | 오류 위험으로 학술 신뢰 부족 |
| 1차→2차 (패키지 전달) | 없음 | **낮음** | BioHub 핵심 해자 |
| 2차 (초고 생성) | ChatGPT/Claude/Gemini | 위협 아님 | BioHub이 이들을 활용하는 구조 |
| 2차→3차 (포맷 변환) | Typst, SciSpace | 낮음 | 한국 학회는 HWP 복붙이 현실 |
| 3차 (최종 편집) | Word/HWP | 위협 아님 | BioHub 영역 밖 |

## 6. 실행 우선순위

```
Phase 완료: 통계 엔진 + Bio-Tools + 자료 작성 템플릿 + 인용 관리 ✅
     ↓
Phase A: AI Export 기능 (구조화된 연구 패키지 내보내기) ← 진행 중
     ↓
Phase B: SOTA 모델 검증 (아래 §6-1 참조)
     ↓
Phase C: 결과 ↔ 논문 섹션 매핑 + 태깅
     ↓
Phase D: 프롬프트 템플릿 (분야별 논문 작성 가이드)
     ↓
Phase E: Publication-ready 그래프 export
     ↓
Phase F: 자체 AI 논문 생성 (API 연동, 아래 §6-2 참조)
```

### 6-1. SOTA 모델 검증 계획 (Phase B)

Package Assembly로 만든 패키지를 실제 SOTA 모델에 넘겨 **논문 초안 품질을 검증**한다.
검증 결과가 좋으면 이를 체계화하여 향후 자체 AI 연동(Phase F)의 기반으로 삼는다.

**검증 대상 모델:**

| 모델 | 특징 | 검증 포인트 |
|------|------|-------------|
| Claude Opus/Sonnet | 긴 컨텍스트, 정밀한 지시 따르기 | 수치 정확도, 구조 준수 |
| GPT-4o / o3 | 가장 범용적, 사용자 다수 보유 | 범용성, 스타일 적응 |
| Gemini 2.5 Pro | 100만 토큰 컨텍스트, Google Scholar 접근 | 대형 패키지 처리, 문헌 보완 |

**검증 프로토콜:**

1. **패키지 준비**: 실제 프로젝트로 Package Assembly 완성 (통계+그래프+문헌+맥락)
2. **초안 생성**: 각 모델에 동일 패키지 전달 → 논문 초안 요청
3. **품질 평가 기준:**
   - 수치 정확도 — 패키지 원본과 초안 내 인용 수치 대조 (F값, p값, 자유도)
   - 구조 준수 — 패키지 태깅(Table 2, Figure 3)이 초안에 정확히 반영되는지
   - 학술 문체 — 분야별 관행 준수 (APA, 수산학 용어 등)
   - Hallucination — 패키지에 없는 데이터/참고문헌 날조 여부
   - 재현성 — 동일 패키지로 반복 생성 시 결과 일관성
4. **결과 기록**: `docs/ai-export-tests/` 폴더에 모델별 결과 저장
5. **개선 반복**: 패키지 구조/프롬프트 템플릿 조정 → 재검증

**검증이 확인해야 할 것:**
- BioHub 패키지가 "충분히 좋은 입력"인지 (garbage in → garbage out 방지)
- 어떤 패키지 구조가 최적인지 (Markdown vs JSON, 섹션 순서, 지시사항 배치)
- 방어적 프롬프트("수치 변경 금지")가 실제로 작동하는지

### 6-2. 자체 AI 논문 생성 로드맵 (Phase F)

SOTA 검증에서 **패키지→초안 파이프라인이 안정화**되면, 외부 모델 없이 BioHub 내부에서 직접 논문을 생성한다.

```
Phase F-1: API 연동 기초
  - Claude/GPT API 키 설정 UI
  - Package → API 호출 → 초안 반환 파이프라인
  - 비용 추정 표시 (토큰 수 기반)

Phase F-2: 자동 논문 생성
  - "패키지에서 초안 생성" 원클릭 버튼
  - 섹션별 생성 (Methods → Results → Discussion 순차)
  - 생성 중 실시간 프리뷰

Phase F-3: 검증 + 편집 루프
  - AI 초안 ↔ 패키지 원본 자동 대조 (수치 검증)
  - 틀린 수치 하이라이트 + 수정 제안
  - DocumentEditor에서 직접 편집 → DOCX/HWPX 내보내기
```

**진입 조건:** Phase B(SOTA 검증) 완료 + 패키지 구조 안정화
**비용 모델:** 사용자 자체 API 키 사용 (BioHub은 중개만) → 운영비 0

## 7. 논문 완성 사이클 — AI 이후 BioHub의 역할

논문은 한 번에 끝나지 않는다. Reviewer 코멘트, 공저자 피드백, 본인 수정으로 **사이클이 반복**된다.

```
     ┌────────────────────────────────────────────────┐
     │              논문 완성 사이클                     │
     │                                                │
┌────┴────┐     ┌─────────┐     ┌──────────┐         │
│ BioHub  │────→│ SOTA AI │────→│ Word/HWP │         │
│ 패키지   │     │ 초고생성  │     │ 최종편집  │         │
└────┬────┘     └─────────┘     └────┬─────┘         │
     │                               │               │
     │    ◀── Reviewer 코멘트 ────────┘               │
     │    ◀── 공저자 피드백 ──────────┘                │
     │    ◀── 본인 수정 ─────────────┘                │
     │                                                │
     └─────────── 반복 ───────────────────────────────┘
```

### BioHub으로 돌아오는 시나리오

| 상황 | BioHub에서 할 일 |
|------|-----------------|
| Reviewer: "분석 조건을 바꿔라" | 재분석 → 새 패키지 export → AI에게 수정 요청 |
| Reviewer: "추가 분석 필요" | 추가 통계 실행 → 패키지에 추가 |
| Reviewer: "Figure 2를 수정해라" | Graph Studio에서 그래프 수정 → 재export |
| Reviewer: "표본 수가 다르다" | 원본 데이터 확인 → 통계 결과 검증 |
| 공저자: "이 해석이 맞나?" | 분석 히스토리에서 원본 결과 확인 |
| 다음 논문 | 같은 데이터로 다른 각도의 패키지 조립 |

### BioHub만 할 수 있는 것 (AI 생성 이후)

1. **검증** — AI가 통계 수치를 정확히 인용했는지 원본 대조 (AI가 F(2,117)=4.23을 4.32로 잘못 쓸 수 있음)
2. **재현성** — "이 논문의 모든 분석을 그대로 다시 돌릴 수 있다"는 보장
3. **버전 관리** — 1차 제출 패키지 vs 수정본 패키지 비교

**BioHub은 논문의 "신뢰할 수 있는 원본 저장소(ground truth)"**다. AI가 만든 글은 바뀌지만, BioHub의 분석 결과는 원본으로 남는다.

## 8. 결론

BioHub의 방향은 적절하며 타이밍도 맞다.

**BioHub은 논문을 쓰는 도구가 아니라, 논문을 쓸 수 있게 만드는 도구다.**

**현재 (Phase A~E): 외부 AI 활용**
- 1차(BioHub): 검증된 분석 + 체계화 → 저비용 Flash Lite로 충분
- 2차(SOTA AI): 사용자의 최고 모델로 초고 생성 → BioHub이 만든 패키지가 품질 결정
- 3차(사람): 학회 템플릿에 맞춰 최종 제출

**향후 (Phase F): 자체 AI 논문 생성**
- SOTA 검증으로 패키지→초안 파이프라인이 안정화되면
- BioHub 내부에서 API 연동으로 직접 초안 생성
- 외부 모델 전환 없이 원클릭 논문 생성 가능

핵심은 **AI Export** — 이 기능이 BioHub의 존재 이유가 된다. 그리고 이 export가 잘 작동하는 것이 곧 자체 AI 논문 생성의 기반이 된다.
